# DEEP-AUDIT RESULTS + FIXES (2026-06-04, 4 parallel investigation agents)

## ✅ FN-14 FIXED (DAW) — root cause + fix
**Root cause (agent-traced):** the STEP 2.7 materialization in `scheduler_core_auto.js` reads the per-bunk slot grid from `window.divisionTimes[grade]._perBunkSlots`. That object is **clobbered mid-run** by the patched `window.loadCurrentDailyData` (`division_times_integration.js:334`, reachable from `_validateWritePlacement`) — which reassigns `window.divisionTimes` and only preserves `_perBunkSlots` under an `_isPerBunk` guard that no longer holds on the 2nd+ in-session gen (after date-load/realtime hydration reassigned `window.divisionTimes`). So the materialization reads `undefined` → `if (!pbs) return` → nothing written → empty schedule, no error. **The codebase already documents this** at `scheduler_core_auto.js:20031-20038` (STEP 4.9), which works around it by reading the durable global `window._perBunkSlots` — the materialization writes were never given the same treatment. **Fix:** all 12 `const pbs = window.divisionTimes?.[grade]?._perBunkSlots;` reads now `= window._perBunkSlots?.[grade] || window.divisionTimes?.[grade]?._perBunkSlots;` (durable copy, reset+populated fresh each run at L16345-16346, survives the clobber; strict-superset fallback). Mirrors the proven L20036 pattern. `node --check` clean. **To live-verify: generate two different dates in ONE page session — both should now apply.**

## ✅ Empty-preview save guard ADDED (DAW) — data-loss prevention
`supabase_schedules.js` wipe guard now also blocks `activitySlotCount === 0` ('all-empty-preview'). Previously the auto-layer preview (bunk keys + unifiedTimes present, but every slot null) had `utLen>0` so it slipped past the `structural-skeleton` guard and could **overwrite a real cloud schedule with all-nulls** — a data-loss path that FN-14's failed gens make reachable. Now a zero-activity payload can't overwrite a real schedule (clearing a day still goes through delete; `options.allowEmpty` still bypasses when intended).

## 🔴 FN-15 — root cause FOUND, fix DEFERRED to a focused live-verified pass
The auto solver's **central write gate** `commitWriteIfLegal` (`auto_solver_engine.js:1265-1317`) — every placement path routes through it — has two gaps: (1) **null-time blind spot** (L1297-1298: occupants whose `_startMin/_endMin` are null are `continue`-skipped → invisible → cross-grade different-activity placements slip past the existing reject), and (2) it **allows same-activity+same-window cross-grade** (L1305 `continue`) with no `same_division` check. The proven mirror is `isFieldAvailableByTime` (auto_solver_engine.js:328-354) + manual `Utils.canBlockFit` (#73, scheduler_core_utils.js:933-940). **Deferred (not bundled with FN-14):** this is the most central gate in the solver; its fix (resolve null times from `window._perBunkSlots[grade][bunk][slotIdx]` + enforce shareType/capacity on the allowance) deserves its own change + live verification so any regression is isolated and testable. Narrow impact (1 field/slot/day) vs FN-14's whole-mode reliability.
